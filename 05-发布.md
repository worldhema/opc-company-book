# 第五章：发布（Release / Deploy）

> 整条流水线里**风险最高**的一环。研发交来一个构建包，发布岗要把它安全地送到所有用户面前。一次错误部署可能瞬间影响全量用户，因此本章基调与前面不同：**AI 只做辅助和加速，关键操作必须人确认，任何上线都要有回滚预案**。AI 原生模式下，发布/运维工程师不再是"手动敲部署命令、盯着日志熬夜的人"，而是"用 AI 自动化流程、智能监控、快速定位异常，自己专注在风险判断和关键决策"的人。

---

## 0. 本章要解决的核心问题

**如何把研发的构建包安全地发布上线？**

发布的本质是"在可控范围内，把变更逐步推给用户，出问题能立刻退回来"。AI 在这里的价值是三件事：

1. **自动化**：CI/CD 流水线、部署脚本、配置校验——把重复且易错的手工操作交给自动化
2. **智能监控**：日志异常检测、指标波动预警、根因初判——从"人盯屏"变成"AI 报警 + 人决策"
3. **快速响应**：出问题时快速定位、生成回滚方案——缩短故障恢复时间

但有一条铁律：**AI 可以按下"建议发布/建议回滚"，真正按下"执行"的必须是人**。发布是不可逆或高代价可逆的动作，不能全托管给 AI。

---

## 1. 角色设定

**岗位名称**：发布 / 运维（DevOps）工程师、发布负责人（AI-Native Release Engineer）

**一句话定位**：你是"上线的指挥官 + 风险的刹车"。AI 负责跑流水线、盯监控、找异常，你负责判断能不能发、要不要退、如何应急。

**人做什么 vs AI 做什么**：

| 环节 | AI 负责（自动化、提速） | 人负责（判断、把关） |
|------|----------------------|---------------------|
| 发布前检查 | 校验配置、依赖、环境差异，生成 checklist | **拍板：这个版本能不能发** |
| 部署 | 执行 CI/CD 流水线、自动化部署脚本 | 触发发布、确认灰度节奏 |
| 灰度放量 | 按策略自动放量、采集各批次指标 | **判断指标是否健康、是否继续放量** |
| 监控 | 实时检测异常、日志聚类、指标预警 | 判断告警是真问题还是误报 |
| 异常处理 | 定位根因、生成回滚/修复方案 | **拍板：回滚还是热修，何时执行** |
| 复盘 | 汇总故障时间线、生成复盘报告初稿 | 判断根因、定改进项 |

**职责边界（红线）**：
- **"执行发布"和"执行回滚"必须人触发**。AI 给建议和方案，人按按钮。
- **生产环境的数据库变更、配置修改、密钥操作**，AI 只能生成脚本供审查，绝不自动执行。
- **灰度是否继续放量、是否全量**，必须由人看指标后决策——AI 的"指标正常"可能漏掉业务层异常。
- 任何发布前，**回滚预案必须就绪**（怎么退、退到哪个版本、数据怎么处理）——没有回滚预案不许上线。

---

## 2. Skills（这个岗位必须具备的 AI 协作能力）

1. **CI/CD 编排提示工程**：能让 AI 生成/优化流水线配置（构建、测试、部署、灰度），并读懂每一步在做什么。
2. **配置与环境校验**：会用 AI 比对不同环境（测试/预发/生产）的配置差异，提前发现"测试好好的，上线就挂"的坑。
3. **监控告警设计**：能让 AI 帮助设计关键指标（错误率、延迟、成功率、资源占用）的监控和告警阈值。
4. **日志分析与根因定位**：会把海量日志和报错喂给 AI 做聚类和根因初判，但对结论保持独立验证。
5. **回滚与应急决策**：懂得预设回滚预案，故障时用 AI 快速生成方案，但由自己做"退还是修"的判断。

---

## 3. AI 工具使用说明（最多 3 个，海外标杆 + 国内平替）

### 工具 ①：CI/CD 流水线 + AI 辅助（自动化部署主力）
- **海外标杆**：GitHub Actions、GitLab CI/CD、Jenkins（配 AI 插件）—— 自动化构建、测试、部署、灰度。
- **国内平替**：**阿里云云效（Flow）、腾讯云 CODING DevOps、华为云 CodeArts、Jenkins 私有部署 + 国产大模型辅助写流水线**。
  - 云效 / CODING：国内主流一站式 DevOps 平台，覆盖代码、流水线、制品、部署、灰度，可对接国内云；
  - 结合通义灵码/DeepSeek 生成和优化流水线脚本、部署配置。
- **典型用法**：让 AI 根据"研发交付说明"生成部署流水线，包含构建、自动化测试、灰度发布、健康检查。

### 工具 ②：监控 + AIOps 智能运维（上线后守护）
- **海外标杆**：Datadog、Grafana、Sentry（错误监控 + AI 分析）—— 指标监控、日志聚合、异常检测、根因分析。
- **国内平替**：**阿里云 ARMS / SLS 日志服务、腾讯云可观测、Prometheus + Grafana（开源自建）、观测云**。
  - 阿里云 ARMS、腾讯云可观测：自带异常检测和智能告警，适合国内云上业务；
  - Prometheus + Grafana：开源自建，配国产大模型做日志分析。
- **典型用法**：AI 实时监控错误率/延迟，异常时自动聚类日志、推送告警并给根因初判。

### 工具 ③：通用大模型（配置校验、根因分析、应急决策辅助、复盘）
- **海外标杆**：Claude / ChatGPT —— 分析配置差异、解读报错、生成回滚方案、写复盘报告。
- **国内平替**：**DeepSeek、通义千问、豆包、Kimi**。
  - DeepSeek：故障根因推理、回滚方案生成强；
  - Kimi/通义：长上下文，适合喂大量日志和配置做分析。
- **典型用法**：把生产报错栈 + 相关日志喂进去，快速定位根因、生成候选修复/回滚方案供人决策。

> **落地建议**：发布岗三件套 = **云效 或 CODING（CI/CD 部署）+ 阿里云可观测 或 Prometheus+Grafana（监控）+ DeepSeek（分析与应急辅助）**。核心是「自动化流水线 + 智能监控 + 通用大模型应急」。

## 4. 工作流程（可直接执行的 SOP）

> 核心链路：**发布前准备 → 灰度部署 → 分批放量 → 实时监控 → 异常回滚/热修 → 全量上线 → 复盘**

### 第 1 步：发布前准备（人拍板"能不能发"）
拿研发交付说明，让 AI 生成发布前检查清单并校验环境。

**提示词模板**：
```
你是一名资深发布工程师。以下是研发交付说明（变更内容、配置依赖、数据库变更、破坏性变更）。请：
1. 生成一份发布前检查清单（checklist）；
2. 重点标注需要人工确认的高风险项（数据库迁移、配置变更、破坏性变更、密钥）；
3. 列出必须准备的回滚预案要素（回滚到哪个版本、数据如何处理、预计耗时）；
4. 指出测试环境与生产环境可能存在的差异风险。
不要遗漏数据库和配置相关的风险项。

研发交付说明：【粘贴】
```
> ⚠️ **回滚预案不就绪，不许上线。** 这是发布岗第一条纪律。

### 第 2 步：灰度部署（AI 执行流水线，人确认节奏）
- 用 CI/CD 流水线自动构建、跑测试、部署到灰度环境。
- 采用**灰度/金丝雀发布**：先放 1% → 5% → 20% → 50% → 100%，每一批都观察后再放下一批。
- **每一次"扩大放量"都由人确认，不自动全量。**

**让 AI 生成灰度流水线的提示词**：
```
基于以下部署需求，生成灰度发布流水线配置（目标平台：【云效/CODING/K8s等】）：
1. 构建 + 自动化测试关卡（测试不过不部署）；
2. 灰度批次：1%→5%→20%→50%→100%，每批之间设人工确认关卡；
3. 每批部署后自动健康检查（错误率、延迟、核心接口成功率）；
4. 健康检查不达标时自动暂停并告警（不自动继续）。
```

### 第 3 步：实时监控（AI 盯屏，人判断告警）
上线后紧盯核心指标，让 AI 做异常检测和日志聚类。

**监控告警设计提示词**：
```
针对本次发布的功能，设计上线监控方案：
1. 核心指标：错误率、接口成功率、P95/P99 延迟、核心业务转化（如下单成功率）；
2. 每个指标的告警阈值和对比基线（与发布前对比）；
3. 需要重点观察的业务日志关键词；
4. 告警分级：哪些需立即人工介入，哪些先观察。
```
> AI 报警后，人要判断：是真问题、还是误报、还是与本次发布无关的既有波动。

### 第 4 步：异常处理（AI 出方案，人拍板"退还是修"）
一旦指标异常，走标准应急流程。

**根因定位 + 方案生成提示词**：
```
生产环境出现异常。以下是报错栈、相关日志、发生时间与本次发布的时间关系。请：
1. 分析最可能的根因（给出可能性排序和依据）；
2. 判断是否与本次发布相关；
3. 给出两个方案：A 立即回滚（步骤+耗时+影响），B 热修（步骤+风险+耗时）；
4. 给出你的建议及理由。
最终决策由人做出，你只提供分析。

报错栈：【粘贴】 日志：【粘贴】 发布时间：【】异常出现时间：【】
```
> **决策原则**：影响面大、根因不明、时间紧 → 优先回滚（先止血）；影响小、根因明确、修复快 → 可考虑热修。**按钮由人按。**

### 第 5 步：全量上线（人确认）
灰度各批次指标健康、观察期通过后，由人确认全量发布。全量后继续观察一段时间再宣布稳定。

### 第 6 步：复盘（AI 写初稿，人定改进项）
无论成功还是出过问题，都做复盘沉淀。

**复盘报告提示词**：
```
基于以下发布记录（时间线、指标变化、发生的问题、处理动作），生成复盘报告初稿：
1. 发布概况（版本、内容、时间、结果）；
2. 时间线（关键节点）；
3. 出现的问题及影响；
4. 处理过程与恢复耗时；
5. 根因分析；
6. 改进项（可落地、有负责人）。
如信息不足以判断根因，明确指出，不要臆测。
```

## 5. 交付物（本环节产出 → 交给"媒体推广"环节）

**上线的产品 + 一份《发布交付说明》**，包含：
1. **已上线的版本**：稳定运行、监控正常的线上产品（用户可用）
2. **发布记录**：版本号、上线时间、灰度节奏、最终结果
3. **正式 Release Note / 更新说明**：面向用户的功能亮点、改进、修复（**推广岗的核心素材**）
4. **上线数据基线**：核心指标的发布后基线（供推广后评估效果、监测流量冲击）
5. **已知问题 / 注意事项**：还有哪些已知小问题、推广时要不要规避

> 这份交付物是下一章"媒体推广"环节的输入——推广岗拿 Release Note 提炼卖点做传播，拿上线数据基线评估推广效果。产品真正稳定上线，才是推广开始的信号——**没上稳就大推，等于把 bug 放大给全网看**。

---

## 附：研发 → 发布 的咬合检查清单（安全上线的核心）

上线前逐条自查，任何一项不满足都不许发：

- [ ] 研发交付说明里的配置、依赖、数据库变更都已在发布 checklist 中确认
- [ ] **回滚预案就绪**（退到哪个版本、数据怎么处理、耗时多久）——没有不许上线
- [ ] 高风险操作（数据库迁移、配置、密钥）已人工审查，不由 AI 自动执行
- [ ] 灰度流水线设有人工确认关卡，不自动全量
- [ ] 监控和告警已配置好，核心指标有发布前基线可对比
- [ ] 应急响应人和决策链清晰（谁判断、谁按回滚按钮）
- [ ] 面向用户的 Release Note 已准备好，交给推广岗
