# 第七章：项目管理（Project & Portfolio Management · 横向统筹层）

> 前六章讲的是"单个产品从需求到推广"的一条纵向流水线。但当公司有**多个产品、多条流水线同时跑**时，谁来协调资源、对齐进度、管控风险、避免撞车？这就是项目管理的位置。它**不是流水线里的一环，而是贯穿全流程、横向协调多条流水线的统筹层**。AI 原生模式下，项目经理不再是"手动更新甘特图、开会催进度、做汇报 PPT 的人"，而是"用 AI 实时聚合多产品状态、自动预警风险、智能调度资源，自己专注在优先级取舍和跨团队决策"的人。

---

## 0. 本章要解决的核心问题

**有了多个产品之后，如何统一管理？**

单产品好办，一条流水线跑到底。多产品的真正难点有四个：

1. **资源抢占**：产品 A、B、C 都要用同一批设计师和工程师，谁先谁后？
2. **进度失真**：多条流水线并行，人工汇总进度慢且滞后，等发现延期已经晚了。
3. **优先级冲突**：公司资源有限，哪个产品该加码、哪个该缓、哪个该砍？
4. **跨产品协同**：多个产品共用组件、数据、基础设施，一处改动牵连全局。

AI 的价值是把"信息聚合 + 状态同步 + 风险识别"这些重复且滞后的工作实时化、自动化，让管理者把精力放在**组合优先级决策**和**跨团队协调**上——这两件事只有人能做。

---

## 1. 角色设定

**岗位名称**：项目经理 / 项目集负责人 / PMO（AI-Native Project & Portfolio Manager）

**一句话定位**：你是"多产品的调度中枢 + 风险的雷达"。AI 负责实时聚合状态、预警风险、生成报告，你负责定组合优先级、调配资源、拍板取舍。

**人做什么 vs AI 做什么**：

| 环节 | AI 负责（聚合、提速、预警） | 人负责（判断、决策） |
|------|--------------------------|---------------------|
| 状态聚合 | 实时汇总各产品/各环节进度、自动更新看板 | 判断状态真实性、识别"报喜不报忧" |
| 进度追踪 | 检测偏差、预测延期、生成进度报告 | 判断延期影响、决定是否干预 |
| 风险预警 | 识别资源冲突、依赖阻塞、进度风险 | **拍板：如何化解、谁让路** |
| 资源调度 | 分析资源占用、给出调配建议 | **决策：资源给哪个产品、加人还是砍范围** |
| 优先级 | 用模型对多产品打分排序 | **拍板：组合优先级、加码/缓/砍** |
| 协同管理 | 梳理跨产品依赖、提醒关联影响 | 协调跨团队、拉齐利益方 |
| 汇报 | 生成周报/月报/管理层汇报初稿 | 定结论、定要资源要什么 |

**职责边界（红线）**：
- **组合优先级和资源分配必须人拍板**。这是公司层面的战略取舍，AI 只提供数据和建议，不替公司做"砍哪个产品"的决定。
- **AI 聚合的进度可能失真**（数据录入滞后、团队报喜不报忧），关键节点要人工核实，不能盲信看板。
- 涉及**人员调动、绩效、组织**的敏感决策，AI 只做辅助分析，人来沟通和决定。

---

## 2. Skills（这个岗位必须具备的 AI 协作能力）

1. **多源信息聚合提示工程**：能让 AI 从各产品的任务系统、文档、沟通记录中提炼统一口径的进度和风险。
2. **风险识别与预测**：会用 AI 从进度数据中提前发现延期信号、资源冲突、依赖阻塞，而不是等问题爆发。
3. **组合优先级建模**：会用 AI 辅助跑加权评分（价值/成本/战略契合/风险），对多产品排序，但对权重和结论人工校准。
4. **资源调度分析**：能让 AI 分析人力占用和瓶颈，给出可执行的调配方案供决策。
5. **结构化沟通与汇报**：会让 AI 把复杂多产品状态浓缩成不同受众（团队/管理层）需要的报告和结论。

---

## 3. AI 工具使用说明（最多 3 个，海外标杆 + 国内平替）

### 工具 ①：项目/任务协作平台（含 AI）（多产品状态中枢）
- **海外标杆**：Jira、Asana、Monday、Linear（含 AI 摘要、进度预测）—— 多项目看板、任务追踪、自动化。
- **国内平替**：**飞书项目（原 MeegO）、腾讯 Tapd、PingCode、ONES、Teambition（阿里）**。
  - 飞书项目 / Tapd / ONES：支持多项目/多产品视图、需求-任务-缺陷全流程追踪、看板和甘特图，可对接研发系统；
  - 结合平台自带或外接的 AI 能力做进度摘要和风险提示。
- **典型用法**：所有产品的任务统一在一个平台，AI 自动生成跨产品进度总览和延期预警。

### 工具 ②：文档/知识协作平台（含 AI）（信息沉淀与对齐）
- **海外标杆**：Notion、Confluence（含 AI）—— 项目文档、会议纪要、决策记录、知识库。
- **国内平替**：**飞书（文档+多维表格+妙记）、腾讯文档、语雀、钉钉**。
  - 飞书多维表格：做多产品组合管理台账（进度、负责人、优先级、风险）极强，可视图+自动化+AI；
  - 飞书妙记：跨产品评审会自动纪要和待办分派。
- **典型用法**：用多维表格建"产品组合管理看板"，AI 汇总各产品状态；会议纪要自动转任务。

### 工具 ③：通用大模型（风险分析、优先级建模、汇报生成）
- **海外标杆**：Claude / ChatGPT —— 数据分析、优先级评分、报告撰写、决策辅助。
- **国内平替**：**DeepSeek、通义千问、豆包、Kimi**。
  - DeepSeek：优先级建模、风险推理、资源冲突分析强；
  - Kimi/通义：长上下文，适合喂多个产品的状态数据做综合分析。
- **典型用法**：把多产品进度、资源占用喂进去，让它识别冲突、给优先级排序、生成管理层汇报。

> **落地建议**：项目管理三件套 = **飞书项目/Tapd/ONES（多产品任务中枢）+ 飞书多维表格（组合管理台账）+ DeepSeek（分析与汇报）**。核心是「统一任务平台 + 组合看板 + 大模型分析」。

## 4. 工作流程（可直接执行的 SOP）

> 核心链路：**统一台账 → 实时聚合状态 → 风险与冲突识别 → 组合优先级决策 → 资源调度 → 跨产品协同 → 汇报与复盘**

### 第 1 步：建立多产品统一台账（管理的地基）
在飞书多维表格/ONES 里建一个覆盖所有产品的组合管理看板，字段统一：

- 产品名 / 负责人 / 当前所处环节（需求-产品-设计-研发-发布-推广，对应第一至六章）
- 优先级 / 战略价值 / 目标上线时间 / 实际进度 / 状态（正常/风险/阻塞）
- 占用资源（哪些人）/ 关键依赖 / 当前风险

> 这个台账是所有 AI 分析的数据源。**口径统一是前提**——各产品用同一套环节定义和状态标准，AI 汇总才有意义。

### 第 2 步：实时聚合状态，生成组合总览
让 AI 定期（如每日/每周）从各产品数据里生成统一总览。

**提示词模板**：
```
你是一名资深 PMO。以下是公司当前所有产品的进度数据。请生成组合总览：
1. 每个产品：当前环节、进度百分比、状态（正常/风险/阻塞）、一句话说明；
2. 整体视图：哪些产品按计划、哪些有风险、哪些阻塞；
3. 需要管理层关注的 Top 3 事项。
基于数据客观呈现，状态存疑处标注【需核实】，不要粉饰。

各产品数据：【粘贴台账】
```

### 第 3 步：风险与资源冲突识别（AI 当雷达）
让 AI 主动扫描延期信号、资源抢占、依赖阻塞。

**提示词模板**：
```
基于以下多产品进度和资源占用数据，识别：
1. 进度风险：哪些产品有延期迹象（依据是什么）；
2. 资源冲突：哪些人/团队被多个产品同时抢占，冲突在什么时间段；
3. 依赖阻塞：哪些产品在等待其他产品/团队，形成阻塞链；
4. 跨产品影响：哪些改动（如共用组件/基础设施）会牵连多个产品。
逐条列出，标注严重程度，并给初步化解建议（最终由人决策）。

数据：【粘贴】
```
> ⚠️ AI 识别冲突，**如何化解、谁让路由人拍板**——这涉及产品间的战略取舍。

### 第 4 步：组合优先级决策（人主导，AI 建模）
资源有限时，用加权评分给多产品排序，AI 出初稿，人校准。

**提示词模板**：
```
对以下产品做组合优先级评分，维度：
- 商业价值（营收/用户增长潜力）
- 战略契合度（是否符合公司方向）
- 紧急度（市场窗口/竞争）
- 投入成本（人力/时间）
- 风险（技术/市场不确定性）
给每个产品各维度打分并加权总分排序，说明理由。
这只是决策参考，最终加码/维持/暂缓/砍由管理层决定。

产品列表及信息：【粘贴】
```
> **拍板"砍哪个、保哪个"是公司战略决策，AI 永远只是参谋。**

### 第 5 步：资源调度（AI 分析，人决策）
基于优先级和冲突，让 AI 给资源调配方案。

**提示词模板**：
```
基于组合优先级和资源冲突，给出资源调配建议：
1. 当前瓶颈资源是什么（哪类角色/哪个人）；
2. 建议如何调配（谁从哪个产品调到哪个产品，影响是什么）；
3. 除了调人，是否有替代方案（砍范围、延期、外包、用AI提效）；
给出2-3套方案及各自利弊，供决策。
```

### 第 6 步：跨产品协同管理
对共用组件、数据、基础设施，用 AI 维护依赖地图，一处改动时提醒所有关联方。用飞书妙记自动记录跨产品评审会的决议和待办，分派到人。

### 第 7 步：汇报与复盘（AI 写初稿，人定结论）
**汇报生成提示词模板**：
```
基于以下组合状态数据，生成【周报/月报/管理层汇报】：
- 面向【团队/管理层】，详略和重点相应调整；
- 整体进展、关键里程碑、主要风险及应对、需要的决策和资源；
- 管理层版本突出：要不要加码/暂缓/砍，要什么资源，结论先行。
数据：【粘贴】
```

## 5. 交付物（横向层的持续产出）

项目管理不产出"一个交给下游的文件"，而是**持续维护整个组织的运转秩序**，核心产出物：

1. **多产品组合看板**：全公司产品状态一屏可见、实时更新
2. **风险预警清单**：延期/冲突/阻塞的提前预警和处置记录
3. **组合优先级决策**：定期更新的产品优先级排序和资源分配方案
4. **跨产品依赖地图**：共用资源和相互影响的关系，改动可追溯
5. **各层级汇报**：团队周报、管理层月报、决策纪要

> **它服务于全流水线**：为需求岗提供"该做哪个产品的需求"的优先级依据；为研发/设计岗协调跨产品的资源和排期；为发布岗对齐多产品的上线节奏（避免同时上线互相抢监控和资源）；为推广岗协调多产品的传播节奏和预算。

---

## 附：多产品管理的核心检查清单

- [ ] 所有产品在同一个台账、同一套环节和状态口径下管理（口径不统一，聚合无意义）
- [ ] 进度数据实时更新，关键节点有人工核实（防看板失真、报喜不报忧）
- [ ] 资源冲突、依赖阻塞有主动预警机制（不等爆发才发现）
- [ ] 组合优先级定期评审，资源向高优先级倾斜（由人决策，非 AI 决定）
- [ ] 跨产品共用资源有依赖地图，改动前评估连带影响
- [ ] 多产品的发布节奏错开或协调（避免同时上线互相干扰）
- [ ] 汇报结论先行、指向决策（要资源/要拍板），不是流水账

---

## 结语：整本手册的闭环

至此，AI 原生公司的运转全景完整了：

- **纵向**（第一至六章）：需求 → 产品 → 设计 → 研发 → 发布 → 推广，一条产品从想法到增长的完整流水线，每一环 AI 跑量提速、人判断把关，交付物层层咬合。
- **横向**（第七章）：项目管理统筹多条流水线，做资源调度、进度对齐、风险管控、组合决策。
- **闭环**：推广收集的用户反馈回流需求岗，整条流水线重新转起，形成增长飞轮。

贯穿全书的一条原则始终不变：**AI 负责跑量、提速、初筛，人负责判断、决策、把关**。AI 原生不是"把公司交给 AI"，而是"让每个人都能指挥 AI，把重复劳动交出去，把判断力留下来"。
