# 第二章：产品（Product）

> 承上启下的枢纽。需求岗交来一份《需求清单 v1》，产品岗要把它变成一份**研发、设计能直接照着做**的产品文档（PRD）。AI 原生模式下，产品经理不再是"埋头写几十页文档的人"，而是"用 AI 把需求快速拆解成方案、跑通逻辑、生成结构化 PRD，自己专注在取舍和决策"的人。

---

## 0. 本章要解决的核心问题

**如何把《需求清单 v1》转化为一份可执行的产品文档？**

一份"可执行"的 PRD，必须能回答研发和设计的三个问题：
1. **做什么**：功能范围、边界（哪些做、哪些明确不做）
2. **怎么做**：业务流程、页面结构、字段规则、异常处理
3. **做到什么程度**：验收标准、数据埋点、上线判定条件

AI 的作用，是把"需求 → 方案 → 结构化文档"这条链路的重复劳动全部接管，让产品经理把精力放在**范围取舍**和**逻辑自洽**上。

---

## 1. 角色设定

**岗位名称**：产品经理（AI-Native Product Manager）

**一句话定位**：你是"方案的架构师 + 范围的守门人"。AI 负责把需求拆成方案、补全逻辑、写文档，你负责定范围、做取舍、保证逻辑不自相矛盾。

**人做什么 vs AI 做什么**：

| 环节 | AI 负责（跑量、提速） | 人负责（判断、把关） |
|------|---------------------|---------------------|
| 方案拆解 | 把用户故事拆成功能模块、生成多个方案备选 | **定范围：哪些做、哪些砍、MVP 是什么** |
| 流程设计 | 生成业务流程、状态流转、异常分支 | 检查流程是否闭环、有无逻辑漏洞 |
| 文档撰写 | 生成 PRD 全文、字段表、验收标准 | 审核准确性、对齐商业目标 |
| 逻辑自检 | 找出文档里的矛盾点、遗漏的异常场景 | 拍板边界情况怎么处理 |
| 数据设计 | 建议埋点方案、成功指标 | 定义什么叫"这个功能成功了" |

**职责边界（红线）**：
- **范围（Scope）必须人定**。AI 会倾向于"什么都做"，导致 MVP 膨胀——砍功能是产品经理的核心价值。
- PRD 里的**业务规则、边界值、异常处理**必须人工逐条确认，AI 生成的"合理默认值"经常想当然。
- 不要把 PRD 一键甩给 AI 生成后不看就发出去——AI 会编造看似专业实则错误的规则。

---

## 2. Skills（这个岗位必须具备的 AI 协作能力）

1. **需求拆解提示工程**：能引导 AI 把一条用户故事拆成"功能模块 + 业务流程 + 页面 + 字段 + 规则"的完整结构，而不是笼统描述。
2. **MVP 收敛能力**：会用 AI 生成多方案，但能用 MoSCoW（Must/Should/Could/Won't）快速砍出最小可行版本。
3. **逻辑自洽审查**：让 AI 扮演"找茬工程师"，主动挑出流程死循环、状态遗漏、异常未覆盖等问题。
4. **PRD 结构化生成**：掌握把 AI 输出直接灌进标准 PRD 模板的方法，产出研发可读的文档。
5. **可量化验收设计**：能让 AI 帮助把模糊目标转成可测的验收标准和埋点指标。

---

## 3. AI 工具使用说明（最多 3 个，海外标杆 + 国内平替）

### 工具 ①：通用大模型（做拆解、方案、写 PRD、逻辑自检）
- **海外标杆**：ChatGPT / Claude —— 长文档结构化、逻辑推演、多轮方案迭代能力最强。
- **国内平替**：**DeepSeek、豆包、Kimi、通义千问**。
  - DeepSeek：逻辑推理强，做方案拆解和逻辑自检最合适；
  - Kimi/通义：长文本强，适合一次性生成完整长篇 PRD；
  - 豆包：日常快速问答、方案头脑风暴。
- **典型用法**：把《需求清单 v1》里的用户故事逐条喂进去，让它拆成功能模块并生成 PRD 草稿。

### 工具 ②：文档协作 + AI 写作平台
- **海外标杆**：Notion AI / Confluence —— PRD 承载、AI 辅助撰写、团队协作评论。
- **国内平替**：**飞书文档（含飞书智能伙伴）/ 腾讯文档 / 语雀**。
  - 飞书文档：PRD 的主战场，内置 AI 可续写、总结、改写，团队实时评论对齐；
  - 语雀：知识库沉淀，适合产品文档归档和版本管理。

### 工具 ③：流程图/原型图生成
- **海外标杆**：Whimsical / Miro（AI 生成流程图、思维导图）。
- **国内平替**：**即时设计 / 摹客 / ProcessOn（AI 生成流程图）+ Mermaid（文本转流程图）**。
  - ProcessOn / Mermaid：把 AI 生成的流程描述直接转成可视化流程图，贴进 PRD；
  - 即时设计：低保真原型快速搭建（高保真交给设计岗）。

> **落地建议**：产品岗三件套 = **DeepSeek（拆解+写 PRD）+ 飞书文档（承载+协作）+ ProcessOn/Mermaid（流程可视化）**。

## 4. 工作流程（可直接执行的 SOP）

> 核心链路：**需求清单 → 功能拆解 → MVP 收敛 → 流程设计 → PRD 生成 → 逻辑自检 → 验收与埋点**

### 第 1 步：读入需求清单，拆解成功能模块
把《需求清单 v1》里的用户故事喂给 AI。

**提示词模板**：
```
你是一名资深产品经理。以下是需求岗交付的用户故事及验收标准。请你：
1. 把每条用户故事拆解成具体的功能模块（一条故事可能对应多个模块）；
2. 每个功能模块标注：核心操作、涉及的页面、关键字段、依赖关系；
3. 指出故事之间的依赖或冲突（比如 A 功能必须先于 B）。
严格基于给定的用户故事，不要新增需求岗没有提出的功能，如需补充请单独列在"建议新增"中。

用户故事如下：
【粘贴《需求清单 v1》的用户故事 + 验收标准】
```

### 第 2 步：MVP 收敛（人主导，最关键的一步）
用 **MoSCoW 模型**砍范围。让 AI 先分类，你再拍板。

**提示词模板**：
```
针对上面拆解出的功能模块，用 MoSCoW 方法分类：
- Must（不做就没法上线的核心）
- Should（重要但可延后一个版本）
- Could（锦上添花）
- Won't（本期明确不做）
给出每个模块的分类和理由。请倾向于把 MVP 做小，只有真正阻断核心流程的才归入 Must。
```
> ⚠️ AI 分类只是初稿。**Must 列表必须由产品经理逐条确认**——这是决定项目大小和上线速度的命门。

### 第 3 步：设计业务流程与状态流转
让 AI 生成主流程 + 异常分支，并输出成 Mermaid 流程图代码。

**提示词模板**：
```
针对【功能模块名】，请设计完整业务流程：
1. 主流程（正常路径，从用户进入到完成）；
2. 异常分支（网络失败、权限不足、数据为空、重复提交等，尽可能穷举）；
3. 涉及的状态流转（如订单：待支付→已支付→已发货→已完成/已取消）；
4. 用 Mermaid flowchart 语法输出流程图代码，我要直接粘贴到文档里。
```

### 第 4 步：生成 PRD 全文（灌入标准模板）
**提示词模板**：
```
基于以上功能拆解、MVP 范围、业务流程，生成一份完整 PRD，严格按以下结构：

# 【产品名/功能名】PRD
## 1. 背景与目标（要解决什么问题，来源需求清单哪条）
## 2. 用户与场景（目标用户、使用场景）
## 3. 功能范围（Must 清单 + 明确不做的 Won't 清单）
## 4. 功能详述（逐个功能：操作流程、页面说明、字段规则、异常处理）
## 5. 业务流程图（Mermaid）
## 6. 数据埋点与成功指标
## 7. 验收标准（可测试、可勾选）
## 8. 非功能需求（性能、安全、兼容性）
## 9. 依赖与风险

字段规则要具体到：字段名、类型、是否必填、校验规则、默认值、错误提示文案。
不确定的地方标注【待确认】，不要编造业务规则。
```
> AI 生成后必须人工过一遍所有【待确认】项和字段规则。

### 第 5 步：逻辑自检（让 AI 当"找茬工程师"）
**提示词模板**：
```
你现在是一名挑剔的研发工程师和测试工程师。请审查以下 PRD，找出：
1. 逻辑矛盾（前后规则打架）；
2. 遗漏的异常场景（没覆盖的边界情况）；
3. 流程死循环或状态无法流转的情况；
4. 字段规则不明确、无法开发的地方；
5. 验收标准中无法测试的模糊表述。
逐条列出问题并给修改建议。宁可多挑，不要放过。

PRD 全文：【粘贴】
```
> 这一步能拦下大量后期返工。研发最恨的就是"PRD 没写清楚，边界情况全靠猜"。

### 第 6 步：验收标准 + 埋点定稿
把验收标准做成可勾选的 checklist，把埋点做成事件表（事件名、触发时机、上报字段），交给研发和数据。

---

## 5. 交付物（本环节产出 → 交给"设计"环节）

一份 **《产品需求文档 PRD v1》**，包含：
1. **功能范围表**：Must / Won't 清单（明确边界）
2. **功能详述**：每个功能的流程、页面、字段规则、异常处理
3. **业务流程图**：Mermaid 或 ProcessOn 导出图
4. **验收标准 checklist** + **埋点事件表**
5. **依赖与风险清单**

> 这份 PRD 是下一章"设计"环节的输入——设计师拿它来做信息架构、交互流程和视觉界面。PRD 里的「功能详述 + 页面说明 + 字段规则」直接决定设计师要画哪些页面、每个页面上有哪些元素。

---

## 附：需求 → 产品 的咬合检查清单

在把 PRD 交给设计岗前，自查这几点，确保上下游对得上：

- [ ] PRD 里每个功能都能追溯到《需求清单 v1》的某条用户故事（不凭空造需求）
- [ ] 《需求清单》里的高优先级需求，都有对应功能覆盖（不漏做）
- [ ] 判定为伪需求的（弃用清单），没有偷偷混进 PRD
- [ ] 每个功能都有明确的验收标准（设计和研发才知道做到什么算完成）
- [ ] 所有【待确认】项都已确认或标注了负责人和截止时间
