# 第四章：研发（Engineering）

> 整条流水线里最能体现"AI 原生"的一环。研发岗要**统一承接两份上游文档——产品岗的 PRD 和设计岗的设计交付包**，跑同一套流程把它们变成可运行、可维护、可交付的代码。AI 原生模式下，工程师不再是"逐行敲代码的人"，而是"用 AI 大规模生成和验证代码，自己专注在架构决策、逻辑正确性和质量把关"的人。

---

## 0. 本章要解决的核心问题

**研发如何承接 PRD 和设计文档，统一进行研发流程？**

难点不在"AI 会不会写代码"，而在**多个工程师、多个 AI 助手协作时，如何不各写各的、不脱离 PRD 和设计规范**。答案是建立一个统一的承接机制：

1. **统一输入**：PRD（做什么、业务规则）+ 设计交付包（长什么样、有哪些状态）汇入同一个任务入口
2. **统一任务拆解**：把两份文档拆成粒度一致、可追溯到需求的开发任务
3. **统一约束**：所有人用同一套项目规范（技术栈、目录结构、组件库、代码规范）喂给 AI，AI 在约束内生成
4. **统一验证**：同一套验收标准（来自 PRD）+ 同一套测试和审查流程

只要这四个"统一"立起来，AI 生成的代码就能拼成一个自洽的产品，而不是一堆风格各异、逻辑打架的碎片。

---

## 1. 角色设定

**岗位名称**：软件工程师 / 研发负责人（AI-Native Engineer）

**一句话定位**：你是"架构的决策者 + 质量的最后一道闸"。AI 负责生成代码、写测试、查 bug，你负责定架构、审逻辑、对齐 PRD 和设计、为质量负责。

**人做什么 vs AI 做什么**：

| 环节 | AI 负责（跑量、提速） | 人负责（判断、把关） |
|------|---------------------|---------------------|
| 需求理解 | 从 PRD/设计稿提取技术要点、生成任务清单 | 判断技术可行性、识别 PRD 里的坑 |
| 架构设计 | 给出技术选型建议、模块划分方案 | **定架构、定技术栈、定分层** |
| 编码 | 生成组件、接口、业务逻辑、样式 | 审代码逻辑、保证符合规范和设计 |
| 测试 | 生成单元测试、边界用例、Mock 数据 | 判断测试是否覆盖真实风险 |
| 代码审查 | 找 bug、找安全隐患、找性能问题 | 拍板是否合并、把关质量红线 |
| 联调对齐 | 比对实现与 PRD/设计的差异 | 判断差异是 bug 还是合理取舍 |

**职责边界（红线）**：
- **架构和技术选型必须人定**。AI 会给"看起来合理"的方案，但不了解你的历史包袱、团队能力和长期维护成本。
- **AI 生成的代码必须人审查后才能合并**。AI 会写出能跑但有安全漏洞、性能陷阱、边界未处理的代码。
- **鉴权、支付、数据库迁移、密钥、隐私数据处理**这类高风险代码，AI 只能辅助，必须人逐行确认。
- 不要让 AI"自由发挥"——不给约束的 AI 会自创技术栈、自造组件、忽略现有代码，导致项目分裂。

---

## 2. Skills（这个岗位必须具备的 AI 协作能力）

1. **上下文喂养能力**：会把项目规范、技术栈、目录结构、现有代码、PRD、设计稿作为上下文喂给 AI，让它在约束内生成，而不是凭空写。
2. **任务拆解提示工程**：能引导 AI 把 PRD+设计稿拆成粒度合适、可独立开发和验证的任务。
3. **代码审查判断力**：能读懂 AI 生成的代码，快速识别逻辑错误、安全隐患、性能问题、不符合规范之处。
4. **测试驱动协作**：会让 AI 先写测试再写实现（或反向补测试），用 PRD 的验收标准约束 AI 的输出。
5. **调试与根因定位**：会把报错、日志、复现步骤喂给 AI 辅助定位，但对根因保持独立判断，不盲信 AI 的"看似合理"的解释。

## 3. AI 工具使用说明（最多 3 个，海外标杆 + 国内平替）

### 工具 ①：AI 编程 IDE / 编码助手（主力生产工具）
- **海外标杆**：Cursor、GitHub Copilot、Claude Code / Windsurf —— 理解整个代码库、多文件编辑、对话式编码、agent 模式自动完成任务。
- **国内平替**：**通义灵码（阿里）、豆包 MarsCode（字节）、CodeGeeX（智谱）、腾讯云 AI 代码助手、文心快码（百度 Comate）**。
  - 通义灵码、MarsCode：支持代码补全、单元测试生成、代码解释、跨文件问答，中文交互友好，可对接国内代码仓库；
  - CodeGeeX：开源可私有化部署，适合对代码不外传有要求的团队。
- **典型用法**：把设计稿组件 + PRD 字段规则喂给它，生成符合项目规范的前端组件和后端接口。

### 工具 ②：代码托管 + AI 代码审查
- **海外标杆**：GitHub（含 Copilot Code Review）、GitLab（含 GitLab Duo）—— PR/MR 流程 + AI 自动审查。
- **国内平替**：**Gitee（码云）、腾讯工蜂、GitLab 私有部署 + CodeGeeX/通义灵码的审查能力**。
  - 用国内代码托管平台管 MR 流程，接入 AI 审查，每次提交自动检查代码规范、潜在 bug、安全隐患。
- **典型用法**：每个 MR 先过 AI 审查（拦低级问题），再由人做关键逻辑审查。

### 工具 ③：通用大模型（架构决策、调试、写文档）
- **海外标杆**：Claude / ChatGPT —— 架构探讨、疑难 bug 分析、技术方案对比。
- **国内平替**：**DeepSeek、通义千问、豆包、Kimi**。
  - DeepSeek：代码推理和疑难 bug 分析强；
  - Kimi/通义：长上下文，适合喂整个模块代码做分析。
- **典型用法**：把报错栈 + 相关代码喂进去分析根因；对比两种技术方案的优劣。

> **落地建议**：研发岗三件套 = **通义灵码 或 MarsCode（主力编码）+ Gitee/工蜂 + AI 审查（协作与质量）+ DeepSeek（架构与调试）**。核心是「主力 IDE 助手 + 代码托管审查 + 通用大模型」这个组合。

## 4. 工作流程（可直接执行的 SOP）

> 核心链路：**统一输入 → 技术评审 → 任务拆解 → 建规范约束 → AI 编码 → 自测+审查 → 联调对齐 → 提测**

### 第 1 步：统一输入 —— 建立研发上下文包
把两份上游文档 + 项目规范汇成一个 AI 可读的"上下文包"，全团队共用。它至少包含：

- **PRD**（功能详述、字段规则、验收标准、埋点）
- **设计交付包**（设计系统/规范、组件库、标注、交互状态）
- **项目规范**（技术栈、目录结构、命名规范、代码风格、已有公共组件和工具函数）

> 在 Cursor / 通义灵码这类工具里，可以把这些放进项目根目录的规范文件（如 `README`、`AGENTS.md`/规则文件），让 AI 每次生成都自动读取——**这是"统一约束"落地的物理载体**。

### 第 2 步：技术评审（人主导）
拿 PRD 和设计稿，团队先过一遍技术可行性，让 AI 辅助识别风险。

**提示词模板**：
```
你是一名资深架构师。以下是 PRD 和设计说明。请从工程角度评审：
1. 技术可行性：有没有难以实现或成本很高的需求/设计；
2. 技术风险：性能、并发、安全、兼容性上的潜在问题；
3. 依赖：需要哪些外部服务/接口/第三方库；
4. 对现有系统的影响：会不会改动到公共模块。
逐条列出，并标注风险等级（高/中/低）。不确定处标注【需人工确认】。

PRD：【粘贴】
设计说明：【粘贴】
```

### 第 3 步：统一任务拆解（保证粒度一致、可追溯）
让 AI 把 PRD+设计拆成标准开发任务，每个任务能追溯到需求。

**提示词模板**：
```
基于 PRD 和设计交付包，把开发工作拆解成任务清单。每个任务包含：
1. 任务名 + 所属功能模块；
2. 对应的 PRD 功能点 / 设计页面（可追溯来源）；
3. 类型（前端/后端/联调）；
4. 输入与输出（依赖哪些接口/数据，产出什么）；
5. 验收标准（直接引用 PRD 的验收条件）；
6. 预估复杂度（高/中/低）和前置依赖任务。
任务粒度控制在 0.5-2 天可完成，太大的拆细。
```
> ⚠️ 这一步的关键是**每个任务都标注了来源和验收标准**——这样多人多 AI 并行开发时，谁都不会偏离 PRD 和设计。

### 第 4 步：在规范约束下用 AI 编码
每写一个任务，把"上下文包 + 该任务的 PRD 片段 + 对应设计稿/标注"一起喂给编码助手。

**前端组件提示词模板**：
```
基于以下上下文生成【组件名】：
- 项目技术栈：【如 Vue3 + TypeScript + 项目UI库】
- 复用现有组件：【列出可复用的公共组件，不要自造】
- 设计规范：颜色/字号/间距用设计系统变量【或贴标注】
- 交互状态：默认/悬停/点击/禁用/加载/空数据/错误（逐一实现，不要漏）
- 字段规则（来自PRD）：【必填、校验规则、错误提示文案】
请遵循项目现有代码风格和目录结构，不要引入未约定的新依赖。
```

**后端接口提示词模板**：
```
基于以下上下文实现【接口名】：
- 业务规则（来自PRD）：【粘贴相关规则和边界条件】
- 输入参数 + 校验规则；输出结构 + 错误码；
- 异常处理：【列出PRD要求的异常场景】
- 遵循项目分层结构和错误处理规范；
- 涉及数据库操作时，说明用到的表和字段，不要臆造表结构。
```

> **红线提醒**：鉴权、支付、数据迁移、密钥相关代码，AI 只做草稿，人逐行审查确认。

### 第 5 步：AI 自测 + 生成单元测试
用 PRD 的验收标准约束测试，让 AI 覆盖边界。

**提示词模板**：
```
为以下代码生成单元测试：
1. 覆盖正常路径；
2. 覆盖 PRD 验收标准里的每一条；
3. 覆盖边界情况：空值、超长、非法输入、并发、异常分支；
4. 用项目现有的测试框架和 Mock 方式。
列出你覆盖了哪些场景、哪些无法自动测试需要人工验证。

验收标准：【粘贴PRD对应条目】
代码：【粘贴】
```

### 第 6 步：提交 MR，过 AI 审查 + 人工审查
- **AI 先审**（拦低级问题）：规范、明显 bug、安全隐患、性能陷阱。
- **人后审**（把关高风险）：核心业务逻辑、架构影响、鉴权/支付/数据安全。

**AI 审查提示词模板**：
```
你是一名严格的代码审查者。审查以下代码变更，找出：
1. 逻辑错误、边界未处理；
2. 安全隐患（注入、越权、敏感信息泄露、未校验输入）；
3. 性能问题（N+1查询、死循环、内存泄漏、不必要的重渲染）；
4. 不符合项目规范之处；
5. 与 PRD/设计的偏差。
逐条列出问题、严重程度、修改建议。宁可多报，不要漏关键风险。
```

### 第 7 步：联调对齐（回到上游校验）
前后端联调后，用 AI 比对"实现 vs PRD/设计"，确保没跑偏。

**提示词模板**：
```
对照 PRD 验收标准和设计稿，逐条检查以下实现是否满足：
- 功能是否都实现了？
- 字段规则、错误提示是否与 PRD 一致？
- 界面还原度、交互状态是否与设计稿一致？
列出符合项、不符合项、待确认项。
```
> 不符合项要区分：是 bug（改代码）、还是合理取舍（回写 PRD/设计留痕）、还是上游文档本身有问题（反馈产品/设计）。

## 5. 交付物（本环节产出 → 交给"发布"环节）

一个 **《可发布的构建包 + 研发交付说明》**，包含：
1. **通过测试和审查的代码**：已合并主干，符合规范
2. **测试报告**：单元测试覆盖情况、验收标准逐条对照结果
3. **构建产物**：可部署的包/镜像
4. **变更说明（Release Note 素材）**：本次改了什么功能、修了什么、有无破坏性变更
5. **配置与依赖说明**：环境变量、外部依赖、数据库变更（如有迁移脚本，重点标注）

> 这份交付物是下一章"发布"环节的输入——发布岗拿它来做上线部署、灰度、监控。研发交付说明里的「变更内容 + 配置依赖 + 破坏性变更」直接决定发布的部署方式和回滚预案。

## 附：设计/产品 → 研发 的咬合检查清单（统一流程的核心）

在提测前自查这几点，确保没脱离上游文档、全团队统一：

- [ ] 每个开发任务都能追溯到 PRD 功能点和设计页面（无凭空开发）
- [ ] PRD 的每条验收标准都有对应的实现和测试覆盖
- [ ] 设计稿的每个状态（空/错/加载/禁用）都已实现（无遗漏态）
- [ ] 全团队用同一套项目规范和上下文包喂 AI（无各写各的、无自创技术栈）
- [ ] AI 生成的代码都经过人工审查后才合并（无未审直接上线）
- [ ] 高风险代码（鉴权/支付/数据/密钥）已人工逐行确认
- [ ] 实现与 PRD/设计的差异都已澄清并留痕（bug 改代码 / 取舍回写文档 / 上游问题反馈）
